3.4 Tasks / Activities Performed

This section provides a detailed account of the frontend development activities carried out during the 10‑week internship for the Sushila Fancy Store e‑commerce website. Each activity is explained with context, approach, challenges encountered, and solutions implemented.

3.4.1 Project Initialization and Environment Setup

The internship began with setting up a robust development environment for the React‑based e‑commerce platform.

-  Technology Selection:  I chose Create React App as the build tool for its simplicity and comprehensive configuration. Later, I integrated custom webpack settings for advanced optimizations.
-  Folder Structure:  I organized the project using a feature‑based structure: `components/`, `pages/`, `hooks/`, `context/`, `services/`, `styles/`, and `utils/`. This separation made the codebase scalable and easy to navigate.
-  Version Control:  I initialized a Git repository on GitHub with a branching strategy: `main` for production, `develop` for integration, and feature branches for each task. I enforced conventional commit messages to keep history clean.
-  Mock API:  Since the backend was not ready, I set up a mock API using `json-server` with a `db.json` file containing product data, categories, and user information. This allowed frontend development to proceed without delays.
-  Environment Variables:  I created a `.env.local` file for API endpoints and other configuration, and documented required variables in `.env.example`.

3.4.2 Component Architecture and Design System

To ensure consistency and reusability, I designed a component library following the atomic design principle.

-  Atoms:  I built foundational components such as `Button`, `Input`, `Icon`, `Heading`, `Text`, and `Container`. Each atom accepted props for variants (e.g., `Button` with primary, secondary, outline), sizes, and custom styling.
-  Molecules:  I combined atoms into more complex components like `ProductCard`, `CartItem`, `NavLink`, and `SearchBar`. For example, `ProductCard` displayed an image, title, price, rating, and an “Add to Cart” button, all arranged in a consistent layout.
-  Organisms:  I created larger sections such as `ProductGrid`, `CartSidebar`, `Header`, and `Footer`. The `ProductGrid` managed the layout of `ProductCard` components and handled loading states.
-  Templates and Pages:  I used templates to structure pages (e.g., `HomeTemplate`, `ShopTemplate`) and implemented pages that imported these templates, with data passed via props.
-  Documentation:  I documented each component using JSDoc comments, specifying props, default values, and usage examples. I also created a Storybook instance for visual testing and documentation.

3.4.3 Product Listing and Filtering System

The product listing page was a core feature, requiring efficient data fetching and state management.

-  Data Fetching:  I used `useEffect` to fetch product data from the mock API and stored it in state. I added loading indicators and error handling to improve user experience.
-  Product Grid:  I implemented a responsive grid using CSS Grid, with columns adapting to screen size (1 column on mobile, 2 on tablet, 4 on desktop). Each product card included a hover effect and quick view option.
-  Filtering:  I built a sidebar with filters for category, price range, and rating. I used `useReducer` to manage filter state, allowing multiple filters to be combined. When filters changed, the product list was updated client‑side without refetching all data.
-  Sorting:  I added a dropdown to sort products by price (low to high, high to low) and by popularity. Sorting was applied to the filtered list using JavaScript’s `sort` method.
-  Pagination:  To avoid overwhelming the user, I implemented pagination with a “Load More” button that appended new products to the grid. I used the Intersection Observer API to enable infinite scrolling as an alternative.

3.4.4 Shopping Cart Development

The shopping cart required global state management to be accessible across multiple pages.

-  Cart Context:  I created a `CartContext` using React’s Context API and a `useReducer` hook to manage cart state. The reducer handled actions: `ADD_ITEM`, `REMOVE_ITEM`, `UPDATE_QUANTITY`, and `CLEAR_CART`.
-  Cart Sidebar:  I developed a slide‑out sidebar that displayed cart items, allowed quantity updates, and showed the subtotal. The sidebar was rendered using a portal to avoid CSS stacking issues, and its open/close state was controlled by context.
-  Local Storage Persistence:  I added a `useEffect` that saved the cart to localStorage whenever it changed, and loaded it on initial render. This ensured the cart persisted across page reloads.
-  Cart Badge:  I added a badge on the cart icon showing the total number of items. This badge updated in real time using the context value.
-  Cart Page:  For users who preferred a full‑page cart, I created a separate `/cart` route with a more detailed view, including item images, subtotals, and a checkout button.

3.4.5 Checkout Flow and Form Validation

The checkout process involved multiple steps and complex validation.

-  Multi‑Step Checkout:  I implemented a three‑step checkout: shipping information, payment method, and order review. Each step was a separate component, with state managed by a parent component.
-  Form Validation:  I used the `react-hook-form` library combined with `yup` schemas for robust validation. Fields were validated in real‑time, and error messages were displayed clearly.
-  Mock Payment Integration:  Since a real payment gateway was not required, I simulated a successful payment by calling a mock API endpoint that returned a fake transaction ID.
-  Order Confirmation:  After checkout, users were shown a confirmation page with their order details and a simulated order number. I also stored the order in localStorage for demonstration.
-  Address Autocomplete:  To enhance user experience, I integrated a free address lookup API that suggested addresses as the user typed (optional feature).

3.4.6 User Authentication Interface

Although the backend authentication was not fully implemented, I built the frontend UI and logic for user authentication.

-  Login and Signup Pages:  I created forms with validation (email format, password strength). I used mock authentication by storing a fake token in localStorage upon successful login.
-  Protected Routes:  I implemented a `PrivateRoute` component that checked for the token and redirected unauthenticated users to the login page.
-  User Profile:  After login, users could access a profile page that displayed their details (mocked) and a list of past orders from localStorage.
-  Logout:  I added a logout button that cleared the token and redirected to the home page.

3.4.7 Responsive Design and Cross‑Browser Compatibility

Ensuring a consistent experience across all devices and browsers was a key requirement.

-  Mobile‑First CSS:  I used CSS Modules with media queries, starting from the smallest breakpoint (320px) and enhancing for larger screens. I defined breakpoints in a central variables file.
-  Layout Components:  I built reusable components like `Grid`, `Flex`, and `Container` that accepted responsive props (e.g., `columns={{ mobile: 1, tablet: 2, desktop: 4 }}`). This simplified responsive layout implementation.
-  Navigation:  The header navigation collapsed into a hamburger menu on mobile. I used `useState` to toggle the menu and CSS transitions for smooth animation.
-  Touch Optimization:  I increased tap targets for buttons and links on mobile to meet accessibility guidelines. I also added `:active` styles for better feedback on touch devices.
-  Cross‑Browser Testing:  I manually tested the site on Chrome, Firefox, Safari, and Edge on both Windows and macOS. I used BrowserStack to test on older versions and on real devices. I fixed browser‑specific issues, such as flexbox bugs in older Safari versions, by using vendor prefixes and fallbacks.

3.4.8 Performance Optimization

To achieve fast load times and smooth interactions, I applied several performance techniques.

-  Image Optimization:  I used the `react-lazy-load-image-component` library to lazy load images below the fold. I also implemented responsive images with `srcset` and served WebP versions where supported.
-  Code Splitting:  I used React’s `lazy` and `Suspense` to split the code for the checkout page and cart sidebar, which are only loaded when needed.
-  Bundle Analysis:  I ran `webpack-bundle-analyzer` to identify large dependencies. I replaced a heavy charting library with a lightweight alternative and removed unused imports.
-  Memoization:  I wrapped components like `ProductCard` with `React.memo` to prevent unnecessary re‑renders when parent components updated.
-  Debouncing Search:  For the search feature, I implemented debouncing to limit API calls (or filter operations) to once per 300ms while the user typed.

3.4.9 Testing and Quality Assurance

As the sole frontend developer on this project, I conducted thorough testing to ensure reliability.

-  Unit Tests:  I wrote unit tests for utility functions (e.g., price formatting, cart calculations) using Jest. I achieved over 85% coverage for core business logic.
-  Component Tests:  I used React Testing Library to test key components, focusing on user interactions. For example, I tested that adding a product to the cart updated the cart badge and sidebar.
-  Integration Tests:  I wrote integration tests for the cart and checkout flow using a mock API to simulate backend calls.
-  Manual Testing:  I executed a detailed test plan covering all features, edge cases, and user scenarios. I documented bugs in GitHub Issues and tracked their resolution.
-  Cross‑Browser and Device Testing:  I repeated manual tests on multiple browsers and devices, ensuring no visual or functional discrepancies.

3.4.10 Accessibility Implementation

I followed WCAG 2.1 guidelines to make the website accessible to all users.

-  Semantic HTML:  I used appropriate elements (`<nav>`, `<main>`, `<button>`) instead of generic divs.
-  ARIA Attributes:  I added `aria-label` to icon‑only buttons and `aria-expanded` to the mobile menu button.
-  Keyboard Navigation:  I tested the entire site using only the keyboard, ensuring all interactive elements were focusable and had visible focus indicators.
-  Screen Reader Testing:  I used NVDA (Windows) and VoiceOver (macOS) to verify that content was read correctly and that dynamic updates (e.g., cart item count) were announced.
-  Color Contrast:  I checked all text and background color combinations against WCAG contrast requirements and adjusted where necessary.
