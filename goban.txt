- Manual Booking Processes: Reliance on phone/email for bookings led to errors, double bookings, and inefficient customer service.

- Administrative Overhead: Staff spent excessive time managing bookings manually rather than focusing on customer service and operations.
- No Payment Integration: Customers couldn't make secure online payments, requiring offline transactions.

This project aimed to address these challenges by developing a comprehensive digital platform that automates booking processes, provides real-time information, and streamlines business operations.

This section provides a detailed account of the full stack development activities carried out during the 10-week internship for the Mount Glacier Trek tourism platform. As a Full Stack Developer, I was responsible for both frontend and backend development, database design, API implementation, payment integration, and deployment. 3.4.1 Project Planning and Requirement Analysis

The internship began with a thorough analysis of the tourism business requirements and technical planning.

-  Stakeholder Meetings:  I met with the client (Mount Glacier Trek) to understand their workflow, booking processes, and pain points. We documented requirements for customer‑facing features (trek browsing, booking, payment) and administrative features (trek management, booking overview, reporting).
-  Technical Stack Selection:  Based on the requirements, I selected the MERN stack (MongoDB, Express.js, React.js, Node.js) for its scalability, developer productivity, and strong ecosystem. MongoDB was chosen for its flexible schema to accommodate varying trek details and booking data.
-  System Architecture Design:  I designed a high‑level architecture: React frontend communicating with a RESTful API built with Node.js/Express, data stored in MongoDB Atlas, authentication using JWT, and payment processing via a third‑party gateway.
-  Project Setup:  I initialized the repository with separate folders for frontend (`client/`) and backend (`server/`). I created a `README.md` with setup instructions and environment variables required.

3.4.2 Database Design and Implementation

A well‑structured database was critical for the booking system.

-  Schema Design:  I designed MongoDB schemas using Mongoose:
  - `User`: fields for name, email, password (hashed), role (customer/admin), profile details.
  - `Trek`: fields for name, description, itinerary, duration, price, difficulty, maxGroupSize, availableDates, images.
  - `Booking`: fields for user reference, trek reference, date, participants, totalPrice, status (pending/confirmed/cancelled), paymentStatus, paymentId.
  - `Payment`: fields for booking reference, amount, method, transactionId, status.
-  Indexing and Performance:  I created indexes on frequently queried fields (`userId`, `trekId`, `status`) to optimize read performance. I also added compound indexes for booking date queries.
-  Data Relationships:  I used Mongoose population to reference related documents (e.g., populating user details in bookings).
-  Data Seeding:  I wrote seed scripts to populate initial trek data and a test admin user for development.

3.4.3 Backend API Development

I built a comprehensive RESTful API using Node.js and Express.

-  API Structure:  I organized routes by resource (`/api/auth`, `/api/users`, `/api/treks`, `/api/bookings`, `/api/payments`). Each route had its own controller and service layer.
-  Authentication Endpoints: 
  - `POST /api/auth/register`: Creates a new user, hashes password with bcrypt, returns JWT.
  - `POST /api/auth/login`: Verifies credentials, returns JWT.
-  User Endpoints: 
  - `GET /api/users/profile`: Returns user profile (protected).
  - `PUT /api/users/profile`: Updates user details.
-  Trek Endpoints: 
  - `GET /api/treks`: Returns list of treks with pagination and filtering (by difficulty, duration, price range).
  - `GET /api/treks/:id`: Returns single trek details.
  - Admin only: `POST`, `PUT`, `DELETE` for trek management.
-  Booking Endpoints: 
  - `POST /api/bookings`: Creates a new booking after checking availability (protected). Updates trek’s availableDates.
  - `GET /api/bookings`: Returns user’s bookings (protected).
  - `GET /api/bookings/:id`: Returns booking details.
  - `PUT /api/bookings/:id/cancel`: Allows user to cancel booking (with refund logic mock).
  - Admin: `GET /api/bookings/admin` to list all bookings, `PUT /api/bookings/:id/status` to update booking status.
-  Payment Endpoints: 
  - `POST /api/payments/initiate`: Initiates payment with gateway, returns payment URL.
  - `POST /api/payments/webhook`: Receives payment status updates from gateway, updates booking and payment statuses.
-  Middleware: 
  - `authMiddleware`: Verifies JWT and attaches user to request.
  - `adminMiddleware`: Checks if user role is admin.
  - `validateRequest`: Uses Joi to validate request body/params before reaching controllers.

3.4.4 Frontend Development

I built the customer‑facing React application with a focus on user experience and responsiveness.

-  Component Architecture:  I structured components by feature: `common/` (Button, Input, Modal), `treks/` (TrekCard, TrekList, TrekDetail), `booking/` (BookingForm, BookingSummary), `user/` (Login, Register, Profile), `admin/` (AdminDashboard, ManageTreks, ManageBookings).
-  State Management:  I used Redux Toolkit for global state (user auth, cart‑like booking summary). I created slices for `auth`, `treks`, `bookings`, and `ui`.
-  Routing:  I implemented client‑side routing with React Router, including protected routes for authenticated pages and admin‑only routes.
-  Data Fetching:  I used RTK Query to define API endpoints and manage caching, loading, and error states. This simplified data fetching and reduced boilerplate.
-  Trek Listing and Detail:  The home page featured a grid of trek cards with images, price, duration. The detail page displayed full itinerary, available dates, and a booking button that opened a modal.
-  Booking Form:  I created a multi‑step booking form that collected number of participants, preferred date (calendar picker with only available dates), and special requests. The form validated inputs and showed real‑time price calculation.
-  User Dashboard:  After login, users could view their bookings, cancel upcoming trips, and edit profile.
-  Admin Dashboard:  I built a separate dashboard accessible only to admins, with tabs for managing treks, viewing all bookings, and viewing payment reports. I used charts (Recharts) to display booking trends.

3.4.5 Payment Gateway Integration

Integrating a payment gateway was crucial for completing the booking process.

-  Gateway Selection:  I used a mock payment gateway (Stripe test mode) for demonstration. I set up a Stripe account and obtained test API keys.
-  Backend Integration: 
  - `POST /api/payments/initiate`: Created a payment intent on Stripe and returned the client secret.
  - `POST /api/payments/webhook`: Listened for Stripe webhook events to confirm payment success/failure.
-  Frontend Integration: 
  - I used Stripe’s React components (`@stripe/react-stripe-js`) to embed a secure payment form on the checkout page.
  - After booking creation, the user was redirected to the payment page where they entered card details (test card 4242 4242 4242 4242).
  - On successful payment, the booking status was updated, and the user saw a confirmation page.
-  Error Handling:  I added fallbacks for network issues and payment failures, displaying user‑friendly messages and allowing retry.

3.4.6 Admin Dashboard and Content Management

The admin dashboard allowed the client to manage the platform without coding.

-  Trek Management:  Admins could create, edit, and delete treks via forms. Images were uploaded to Cloudinary, and URLs stored in the database.

3.4.7 Authentication and Security

I implemented robust security measures to protect user data and the platform.

-  JWT Tokens:  Generated JWT with expiration; stored in HTTP‑only cookies (backend) to prevent XSS attacks. I also implemented refresh token rotation for better security.
-  Input Validation:  Used Joi to validate all incoming request data, preventing injection attacks.
-  Rate Limiting:  Added `express-rate-limit` to limit API requests per IP, preventing brute force attacks.
-  Helmet:  Used Helmet.js to set secure HTTP headers.
-  CORS:  Configured CORS to allow only the frontend domain.
-  MongoDB Security:  Used connection string with credentials and network access rules to restrict access.
